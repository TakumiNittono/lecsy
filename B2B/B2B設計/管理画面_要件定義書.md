# Lecsy B2B 管理画面 — 要件定義書

**作成日:** 2026-04-01
**対象:** Web ダッシュボード（Next.js 14.2 / Supabase / Stripe）
**目的:** 語学学校・教育機関向けB2B管理画面の構築

---

## 1. 現状整理

### 既存テーブル（Supabase PostgreSQL 15）
| テーブル | 用途 | 備考 |
|---------|------|------|
| transcripts | 文字起こしデータ | user_id で所有者管理、RLS有効 |
| summaries | AI要約・試験対策 | transcript_id に紐づく |
| subscriptions | 個人サブスク管理 | Stripe連携、status: free/active/canceled/past_due |
| usage_logs | AI機能使用ログ | summarize / exam_mode |
| rate_limit_logs | レート制限 | user_id + action + created_at |
| reports | バグ報告 | 管理者のみ閲覧可 |

### 既存Webルート
| パス | 用途 | 認証 |
|------|------|------|
| / | ランディングページ | 不要 |
| /login | ログイン | 不要 |
| /app | 個人ダッシュボード | 必要 |
| /app/t/[id] | 文字起こし詳細 | 必要 |
| /app/reports | バグ報告管理（管理者のみ） | 必要 + ホワイトリスト |

### 既存技術スタック
- **フレームワーク:** Next.js 14.2（App Router）
- **スタイル:** Tailwind CSS 3.4
- **認証:** Supabase Auth（Google / Apple OAuth）
- **DB:** Supabase PostgreSQL 15（RLS有効）
- **決済:** Stripe（checkout session / billing portal / webhook）
- **Edge Functions:** save-transcript, summarize, stripe-webhook, delete-account, submit-report
- **ホスティング:** Vercel
- **ポート:** 3020（dev/prod）

### 組織機能の現状
**ゼロ。** 組織・チーム・招待・ロール管理に関するテーブル、API、UIは一切存在しない。

## 2. 新規DB設計

### 2.1 organizations テーブル
```sql
CREATE TABLE organizations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  slug TEXT NOT NULL UNIQUE,  -- URL用（例: ohla-orlando）
  type TEXT NOT NULL DEFAULT 'language_school'
    CHECK (type IN ('language_school', 'university_iep', 'college', 'corporate')),
  plan TEXT NOT NULL DEFAULT 'starter'
    CHECK (plan IN ('starter', 'growth', 'enterprise')),
  max_seats INTEGER NOT NULL DEFAULT 50,
  logo_url TEXT,
  stripe_customer_id TEXT,
  stripe_subscription_id TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- インデックス
CREATE INDEX idx_organizations_slug ON organizations(slug);
CREATE INDEX idx_organizations_stripe ON organizations(stripe_customer_id);

-- updated_at トリガー
CREATE TRIGGER update_organizations_updated_at
  BEFORE UPDATE ON organizations
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

### 2.2 organization_members テーブル
```sql
CREATE TABLE organization_members (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  role TEXT NOT NULL DEFAULT 'student'
    CHECK (role IN ('owner', 'admin', 'teacher', 'student')),
  joined_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(org_id, user_id)
);

-- インデックス
CREATE INDEX idx_org_members_org ON organization_members(org_id);
CREATE INDEX idx_org_members_user ON organization_members(user_id);
CREATE INDEX idx_org_members_role ON organization_members(org_id, role);
```

### 2.3 organization_invites テーブル
```sql
CREATE TABLE organization_invites (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  email TEXT NOT NULL,
  role TEXT NOT NULL DEFAULT 'student'
    CHECK (role IN ('admin', 'teacher', 'student')),
  token TEXT NOT NULL UNIQUE DEFAULT encode(gen_random_bytes(32), 'hex'),
  invited_by UUID NOT NULL REFERENCES auth.users(id),
  accepted BOOLEAN NOT NULL DEFAULT false,
  expires_at TIMESTAMPTZ NOT NULL DEFAULT (now() + interval '7 days'),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- インデックス
CREATE INDEX idx_org_invites_token ON organization_invites(token);
CREATE INDEX idx_org_invites_email ON organization_invites(email);
CREATE INDEX idx_org_invites_org ON organization_invites(org_id);
```

### 2.4 RLSポリシー
```sql
-- organizations
ALTER TABLE organizations ENABLE ROW LEVEL SECURITY;

-- メンバーなら閲覧可
CREATE POLICY "org_select_member" ON organizations FOR SELECT USING (
  EXISTS (
    SELECT 1 FROM organization_members
    WHERE organization_members.org_id = organizations.id
    AND organization_members.user_id = (SELECT auth.uid())
  )
);

-- owner/admin のみ更新可
CREATE POLICY "org_update_admin" ON organizations FOR UPDATE USING (
  EXISTS (
    SELECT 1 FROM organization_members
    WHERE organization_members.org_id = organizations.id
    AND organization_members.user_id = (SELECT auth.uid())
    AND organization_members.role IN ('owner', 'admin')
  )
);

-- organization_members
ALTER TABLE organization_members ENABLE ROW LEVEL SECURITY;

-- 同組織のメンバーなら閲覧可
CREATE POLICY "members_select" ON organization_members FOR SELECT USING (
  EXISTS (
    SELECT 1 FROM organization_members AS om
    WHERE om.org_id = organization_members.org_id
    AND om.user_id = (SELECT auth.uid())
  )
);

-- owner/admin のみ追加・削除可
CREATE POLICY "members_insert_admin" ON organization_members FOR INSERT WITH CHECK (
  EXISTS (
    SELECT 1 FROM organization_members AS om
    WHERE om.org_id = organization_members.org_id
    AND om.user_id = (SELECT auth.uid())
    AND om.role IN ('owner', 'admin')
  )
);

CREATE POLICY "members_delete_admin" ON organization_members FOR DELETE USING (
  EXISTS (
    SELECT 1 FROM organization_members AS om
    WHERE om.org_id = organization_members.org_id
    AND om.user_id = (SELECT auth.uid())
    AND om.role IN ('owner', 'admin')
  )
);

-- organization_invites
ALTER TABLE organization_invites ENABLE ROW LEVEL SECURITY;

-- 同組織のadmin以上が閲覧可
CREATE POLICY "invites_select_admin" ON organization_invites FOR SELECT USING (
  EXISTS (
    SELECT 1 FROM organization_members
    WHERE organization_members.org_id = organization_invites.org_id
    AND organization_members.user_id = (SELECT auth.uid())
    AND organization_members.role IN ('owner', 'admin')
  )
);

-- admin以上が招待作成可
CREATE POLICY "invites_insert_admin" ON organization_invites FOR INSERT WITH CHECK (
  EXISTS (
    SELECT 1 FROM organization_members
    WHERE organization_members.org_id = organization_invites.org_id
    AND organization_members.user_id = (SELECT auth.uid())
    AND organization_members.role IN ('owner', 'admin')
  )
);
```

### 2.5 既存テーブルへの影響
**transcripts テーブルは変更しない。** 組織メンバーの使用状況は `organization_members.user_id` 経由で `transcripts` を JOIN して取得する。データ所有権は個人ユーザーのまま維持。

## 3. ロール定義

| ロール | 権限 |
|--------|------|
| **owner** | 全権限。組織の作成者。Stripe課金の管理。組織の削除。 |
| **admin** | メンバー管理（招待・削除）。使用統計の閲覧。設定変更。ownerの削除不可。 |
| **teacher** | 自クラスの学生の使用状況閲覧。自分のtranscript管理。 |
| **student** | 自分のtranscript管理のみ。管理画面アクセス不可。 |

### 4.2 各画面の詳細

#### `/org/new` — 組織作成
**目的:** 新しい組織を作成する（営業デモ時にも使用）

**フォーム項目:**
- 組織名（必須、最大100文字）
- 組織タイプ（ドロップダウン: language_school / university_iep / college / corporate）
- slug（自動生成 + 手動編集可、英数字とハイフンのみ、最大50文字）

**処理フロー:**
1. バリデーション
2. `organizations` テーブルにINSERT
3. 作成者を `organization_members` に `role: 'owner'` でINSERT
4. `/org/[slug]` にリダイレクト

**UI:**
- シンプルなカード型フォーム
- 作成ボタン1つ

---

#### `/org/[slug]` — 組織ダッシュボード
**目的:** 組織の概要を一目で把握する

**表示内容:**

1. **ヘッダー**
   - 組織名
   - プランバッジ（Starter / Growth / Enterprise）
   - 座席数（使用中 / 上限）

2. **統計カード（4枚）**
   - 総メンバー数（teacher + student）
   - 今月の総録音数（全メンバーのtranscript count）
   - 今月の総録音時間（全メンバーのduration合計、hh:mm形式）
   - アクティブユーザー数（過去7日間にtranscriptを作成したメンバー数）

3. **最近のアクティビティ**
   - 直近10件のtranscript（メンバー名、タイトル、日時、時間、言語）
   - 「もっと見る」→ `/org/[slug]/usage` へ

4. **クイックアクション**
   - 「メンバーを招待」ボタン → `/org/[slug]/invite`
   - 「使用統計を見る」ボタン → `/org/[slug]/usage`

**データ取得:**
```sql
-- 統計クエリ例
SELECT
  COUNT(DISTINCT om.user_id) AS total_members,
  COUNT(t.id) AS monthly_transcripts,
  COALESCE(SUM(t.duration), 0) AS monthly_duration,
  COUNT(DISTINCT CASE
    WHEN t.created_at >= now() - interval '7 days'
    THEN om.user_id
  END) AS active_users
FROM organization_members om
LEFT JOIN transcripts t ON t.user_id = om.user_id
  AND t.created_at >= date_trunc('month', now())
WHERE om.org_id = $1;
```

#### `/org/[slug]/members` — メンバー管理
**目的:** メンバーの一覧表示・ロール変更・削除

**表示内容:**

1. **メンバーテーブル**
   - 列: 名前（auth.usersから）、メール、ロール、参加日、最終録音日
   - ソート: ロール順（owner → admin → teacher → student）→ 参加日順
   - 検索: 名前・メールでフィルタ

2. **アクション（owner/adminのみ）**
   - ロール変更（ドロップダウン: admin / teacher / student）
     - ownerロールは変更不可
     - 自分自身のロール変更不可
   - メンバー削除（確認ダイアログ付き）
     - ownerは削除不可
     - 自分自身の削除不可
   - 「メンバーを招待」ボタン

3. **座席情報**
   - 「X / Y 座席使用中」の表示
   - 上限に達した場合: 「座席上限に達しました。プランをアップグレードしてください。」

**API:**
- `PATCH /api/org/[slug]/members/[memberId]` — ロール変更
- `DELETE /api/org/[slug]/members/[memberId]` — メンバー削除

#### `/org/[slug]/invite` — メンバー招待
**目的:** メールアドレスで新メンバーを招待する

**フォーム項目:**
- メールアドレス（必須、複数入力可: カンマ区切りまたは改行区切り、最大20件/回）
- ロール（ドロップダウン: teacher / student、デフォルト: student）

**処理フロー:**
1. メールアドレスのバリデーション（形式チェック）
2. 既存メンバーの重複チェック
3. 座席上限チェック（現メンバー数 + 招待数 ≤ max_seats）
4. `organization_invites` にINSERT（token自動生成、7日有効）
5. 招待リンク生成: `https://lecsy.app/invite/[token]`
6. 画面に招待リンクを表示（コピーボタン付き）

**補足:**
- MVP段階ではメール自動送信は行わない
- 管理者が招待リンクをコピーして、メール/Slack/LINEで送る運用
- 将来的にSupabase Edge Function + Resend等でメール自動送信を追加

**UI:**
- テキストエリア（複数メール入力）
- ロール選択ドロップダウン
- 「招待リンクを生成」ボタン
- 生成後: リンク一覧 + コピーボタン

**ペンディング招待一覧:**
- 未承諾の招待リスト（メール、ロール、有効期限、ステータス）
- 招待キャンセルボタン（DELETE）

---

#### `/org/[slug]/usage` — 使用統計
**目的:** 組織全体の使用状況を可視化する

**表示内容:**

1. **期間選択**
   - 今週 / 今月 / 先月 / カスタム範囲
   - デフォルト: 今月

2. **サマリーカード**
   - 総録音数
   - 総録音時間
   - アクティブユーザー数 / 全メンバー数
   - 平均録音時間/ユーザー

3. **メンバー別使用状況テーブル**
   - 列: 名前、ロール、録音数、総時間、最終使用日、使用言語
   - ソート可能（各列ヘッダークリック）
   - CSVエクスポートボタン

4. **言語別統計**
   - 各言語の使用回数（横棒グラフ or テーブル）
   - 12言語の分布を可視化

5. **日別推移**（将来）
   - 日別の録音数推移（折れ線グラフ）
   - MVPでは省略可

**データ取得:**
```sql
-- メンバー別使用状況
SELECT
  om.user_id,
  au.raw_user_meta_data->>'full_name' AS name,
  au.email,
  om.role,
  COUNT(t.id) AS transcript_count,
  COALESCE(SUM(t.duration), 0) AS total_duration,
  MAX(t.created_at) AS last_used,
  array_agg(DISTINCT t.language) FILTER (WHERE t.language IS NOT NULL) AS languages
FROM organization_members om
JOIN auth.users au ON au.id = om.user_id
LEFT JOIN transcripts t ON t.user_id = om.user_id
  AND t.created_at >= $2  -- 期間開始
  AND t.created_at < $3   -- 期間終了
WHERE om.org_id = $1
GROUP BY om.user_id, au.raw_user_meta_data, au.email, om.role
ORDER BY transcript_count DESC;
```

#### `/org/[slug]/billing` — 課金管理
**目的:** B2Bプランの管理・Stripe連携

**表示内容:**

1. **現在のプラン**
   - プラン名（Starter / Growth / Enterprise）
   - 月額料金
   - 座席上限
   - 次回請求日
   - ステータス（active / past_due / canceled）

2. **プラン変更**
   - Starter → Growth → Enterprise へのアップグレード
   - Stripe Checkout Session経由

3. **請求履歴**
   - Stripe Customer Portal へのリンク
   - 直近の請求一覧（Stripe APIから取得）

4. **支払い方法**
   - Stripe Customer Portal で管理

**Stripe連携:**
- B2B用の新しいStripe Price IDが必要
  - `price_starter`: $99/月
  - `price_growth`: $249/月
  - `price_enterprise`: カスタム
- checkout session 作成時に `metadata.org_id` を含める
- webhook で `organization` の `stripe_customer_id` / `stripe_subscription_id` を更新

#### `/invite/[token]` — 招待受諾
**目的:** 招待リンクからメンバーとして参加する

**処理フロー:**
1. token で `organization_invites` を検索
2. バリデーション:
   - tokenが存在するか
   - 有効期限内か（expires_at > now()）
   - 既に承諾済みでないか
3. 未ログインの場合: ログインページへリダイレクト（next=/invite/[token]）
4. ログイン済みの場合:
   - 組織名とロールを表示
   - 「参加する」ボタン
5. 参加処理:
   - `organization_members` にINSERT
   - `organization_invites` の `accepted` を true に更新
   - ユーザーの `subscriptions` を `active`（Pro相当）に更新
   - `/org/[slug]` にリダイレクト

## 7. 既存機能への影響

### 7.1 サブスクリプション連携
組織メンバー（teacher/student）は、組織のプランが `active` である限り **自動的にPro機能を利用可能** にする。

**判定ロジック変更（summarize Edge Function）:**
```
現在: ホワイトリスト OR subscription.status = 'active'
変更: ホワイトリスト OR subscription.status = 'active' OR 所属組織のplan != null
```

具体的には:
1. ユーザーの `organization_members` を確認
2. 所属する `organizations` の `plan` が `starter` / `growth` / `enterprise` のいずれかなら Pro 扱い
3. 個人サブスクと組織サブスクは独立（どちらかがアクティブならPro）

### 7.2 iOSアプリへの影響
**MVPでは変更しない。** iOSアプリは個人ユーザーとして動作し続ける。組織への帰属はWeb側で管理。iOSユーザーが組織に招待された場合、同じSupabase Auth UIDで紐づくため、Pro機能は自動的に解放される。

## 8. 画面デザイン方針

### スタイルガイド
- **既存のTailwindクラスを踏襲**（globals.cssのアニメーション・カラー変数を再利用）
- **配色:** 白背景 + グレーボーダー + アクセントカラー（青系）
- **フォント:** システムフォント（既存設定を継続）
- **レスポンシブ:** モバイルファースト（PC管理画面だが、営業デモ時にiPadで見せる可能性あり）
- **コンポーネント設計:** 既存の components/ にフラットに追加（新ディレクトリ不要）

### 共通UIパターン
- **統計カード:** 白背景 + 影 + アイコン + 数値 + ラベル（既存SubscriptionCardと同じスタイル）
- **テーブル:** ストライプ行（奇数行に薄いグレー背景）
- **ボタン:** Primary（青）/ Secondary（グレー）/ Danger（赤）
- **モーダル:** 確認ダイアログ（削除等の破壊的操作時）
- **トースト:** 既存のToastコンポーネントを再利用

## 11. セキュリティ要件

| 項目 | 対策 |
|------|------|
| 認可 | 全APIで `requireOrgRole` による権限チェック |
| RLS | 全テーブルでRLS有効。直接DBアクセスでもデータ漏洩なし |
| CSRF | 既存のOrigin/Refererチェックを全APIに適用 |
| 入力検証 | slug: `/^[a-z0-9-]+$/`、名前: 最大100文字、メール: 形式チェック |
| 招待トークン | 256bit暗号的ランダム（`gen_random_bytes(32)`）、7日有効期限 |
| レート制限 | 招待作成: 50/時間、組織作成: 5/時間 |
| ownerの保護 | ownerのロール変更・削除は不可 |

## 12. テスト観点

| テスト | 内容 |
|--------|------|
| 認可テスト | 各ロールで許可/拒否される操作の確認 |
| 座席制限 | max_seats超過時に招待がブロックされるか |
| 招待フロー | リンク生成 → 受諾 → メンバー追加の一連の流れ |
| 有効期限 | 期限切れの招待トークンが拒否されるか |
| 重複チェック | 既存メンバーへの再招待が適切にハンドリングされるか |
| 組織削除 | CASCADE削除でメンバー・招待が全削除されるか |
| Pro連携 | 組織メンバーがPro機能を利用できるか |
| RLS | 別組織のデータにアクセスできないか |
