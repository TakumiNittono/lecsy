# SNS 自動化 — 運用手順

> 親: [[MOC_SNS]] / 決定は [[SNS/自動化_決定ログ]]
> コード: `lecsy-video/scripts/sns-{run,generate,guardrail,post-x}.mjs`

## 1. 初期セットアップ (最初の 1 回)

### X Developer 側 (あなた)
1. https://developer.x.com にログイン (@takumi_global)
2. **クレジット**: ダッシュボードから最低 $5 + 自動チャージ有効化
3. **アプリ作成**: 左サイドバー「アプリ」→ Create App
   - Name: `lecsy-sns-auto`
   - App permissions: **Read and Write** ← 最重要
   - App type: **Web App, Automated App or Bot**
   - Callback URI: `http://localhost:3000/callback` (使わないが必須)
   - Website URL: `https://lecsy.app`
4. **Keys and tokens** タブで以下を取得:
   - API Key / API Key Secret
   - Access Token / Access Token Secret (Generate ボタン)

### OpenAI API 側 (あなた)
1. https://platform.openai.com/api-keys で API key 作成 (既存キー流用可)
2. 使用モデル: `gpt-4o` (.env.local の `OPENAI_MODEL` で切替可)

### ローカル (一度だけ)
```bash
cd "/Users/takuminittono/Desktop/iPhone app/lecsy/sns-auto"

# .env.local に追記 (既存の HEYGEN_API_KEY と一緒に)
cat >> .env.local <<'EOF'
X_API_KEY=...
X_API_KEY_SECRET=...
X_ACCESS_TOKEN=...
X_ACCESS_TOKEN_SECRET=...
OPENAI_API_KEY=...
OPENAI_MODEL=gpt-4o
SNS_AUTO_DISABLED=false
EOF

# deps は既に install 済
```

## 3. GitHub Actions Cron (本番)

`.github/workflows/sns-auto.yml` が cron を管理。4 本:
- 04:00 JST — 情報収集 (sns-collect.mjs)
- 08:00 JST — 朝投稿 (P1 数字)
- 12:30 JST — 昼投稿 (P3 技術)
- 21:00 JST — 夜投稿 (P2 OPT)

**手動テスト:** Actions タブ → SNS Auto Post → Run workflow → action を選択

### GHA に secrets を登録 (最初に 1 回)

```bash
cd "/Users/takuminittono/Desktop/iPhone app/lecsy"

# .env.local の値を GHA secrets に流し込む
gh secret set X_HANDLE              --body "$(grep ^X_HANDLE=              sns-auto/.env.local | cut -d= -f2-)"
gh secret set X_API_KEY             --body "$(grep ^X_API_KEY=             sns-auto/.env.local | cut -d= -f2-)"
gh secret set X_API_KEY_SECRET      --body "$(grep ^X_API_KEY_SECRET=      sns-auto/.env.local | cut -d= -f2-)"
gh secret set X_ACCESS_TOKEN        --body "$(grep ^X_ACCESS_TOKEN=        sns-auto/.env.local | cut -d= -f2-)"
gh secret set X_ACCESS_TOKEN_SECRET --body "$(grep ^X_ACCESS_TOKEN_SECRET= sns-auto/.env.local | cut -d= -f2-)"
gh secret set OPENAI_API_KEY        --body "$(grep ^OPENAI_API_KEY=        sns-auto/.env.local | cut -d= -f2-)"
gh secret set SLACK_WEBHOOK_URL     --body "$(grep ^SLACK_WEBHOOK_URL=     sns-auto/.env.local | cut -d= -f2-)"

# 確認
gh secret list
```

### 一時停止 (kill switch, GHA 側)

```bash
# Repo Variables で SNS_AUTO_DISABLED=true にする
gh variable set SNS_AUTO_DISABLED --body "true"

# 再開
gh variable set SNS_AUTO_DISABLED --body "false"
# または delete
gh variable delete SNS_AUTO_DISABLED
```

### Cron の時刻変更

`.github/workflows/sns-auto.yml` の `cron:` 行を編集して push。
JST -> UTC は -9 時間 (例: JST 08:00 = UTC 前日 23:00)。

## 4. tone 調整の流れ

- Claude の出力が気に食わなかったら `_Vault/SNS/tone_sample.md` を更新
- 具体例 (過去の自分のツイート or 好きな人) を貼る ほど精度上がる
- 修正したら dry-run で 5 件生成して確認 → OK なら live に戻す

## 5. 在庫の補充

- `_Vault/SNS/在庫_マップ.md` の候補が 5 件未満になったら追加
- 投稿したら `最終投稿日` を埋める (sns-run.mjs は現状まだ埋めない — 次フェーズで自動化)
- 同じ sourceNote は 30 日間は再選ばれない (sns-run.mjs の `alreadyPosted` ガード)

## 6. Kill switch

### 即時停止
```bash
# ローカル cron の場合
echo "SNS_AUTO_DISABLED=true" >> .env.local

# Vercel の場合
vercel env add SNS_AUTO_DISABLED production
# value に "true" を入力
vercel --prod   # 反映
```

### 発動条件 (いずれかで即発動)
- 炎上の兆し (リプ > 通常の 3 倍 / block 急増)
- 事実誤認を発見
- メンタル的に SNS が負担になった時
- engagement が 3 週連続で低下

### 発動後
1. 過去 7 日分の `_Vault/SNS/公開済/` を読み返す
2. tone_sample.md を更新 / NG ワード追加 / 在庫候補を整理
3. dry-run を 3 日間走らせて出力チェック
4. 問題なければ `SNS_AUTO_DISABLED=false` に戻して再開

## 7. Slack 通知

### 仕込み (1 回)

1. https://api.slack.com/apps → 対象アプリ (or 新規作成) → **Incoming Webhooks**
2. 通知チャンネルを決める (推奨: `#lecsy-sns-alerts` を新規作成)
3. 「Add New Webhook to Workspace」→ URL コピー
4. `.env.local` の `SLACK_WEBHOOK_URL=` に貼る
5. テスト: `npm run sns:notify:test` → チャンネルに投稿が届けば成功

### 通知される条件

| レベル | 条件 | 対応 |
|---|---|---|
| 🚨 critical | 認証エラー (X/OpenAI 401) / クレジット不足 (402) | キー再発行 or 課金追加 |
| ❌ error | 予期しない throw / OpenAI その他エラー | ログ確認 |
| ⚠️ warning | レート制限 (429) / 重複投稿 (403) / guardrail 2件以上 block / 在庫枯渇 / kill switch 発動 | 翌日まで待つ or 在庫追加 |
| ✅ success | 投稿成功 (デフォルトでは送らない) | `SLACK_NOTIFY_SUCCESS=true` で ON |

### 黙らせたい時
```bash
# .env.local に追加
SLACK_NOTIFY=false          # 全部 OFF
SLACK_NOTIFY_SUCCESS=false  # 成功通知だけ OFF (デフォ)
```

## 8. 異常パターンと対応

| 症状 | 原因候補 | 対応 |
|---|---|---|
| cron ログに `guardrail BLOCKED` 連発 | tone 不一致 or NG ワード hit | tone_sample.md 更新 / NG リスト見直し |
| X API 401 | Access Token の write 権限なし | X Developer でトークン再発行 (Read and Write) |
| X API 403 "duplicate content" | 同じ文面を短時間で再投稿 | guardrail に hash 重複検出を追加 (未実装) |
| OpenAI 429 | レート上限 | cron の実行時刻を分散 / max_tokens 削減 |
| 投稿はされたが内容がおかしい | tone prompt が弱い | tone_sample.md に "ダメな例" セクションを追加 |
