# Lecsy B2B 統合要件定義書 v4（エンタープライズ本気版）

**作成日:** 2026-04-06
**ステータス:** 確定版（v3 を継承し B2B を本気仕様へ昇格）
**継承元:** `doc/00_統合要件定義_v3.md`
**対象顧客:** 語学学校 / IEP / 大学 / 企業研修部門（年契 $3K〜$100K+）

> **方針アップデート:** v3 までの「営業 MVP」を脱し、**RFP に応えられる B2B SaaS 仕様**に引き上げる。教育機関の調達部・情シス・法務・コンプラ部門のレビューを通せる水準を目指す。
>
> **継続するもの（v3 から不変）:**
> - iOS をプライマリ、Web をセカンダリの 2 面 1 バックエンド
> - メール一致による pending → active 自動昇格
> - スーパー管理者経由の owner 付与
> - 招待リンク方式は不採用
> - DB マイグレーション順序 M6→M5→M1〜M4→M7〜M13

## 0. B2B 本気化の判断基準

調達側が「**社内稟議を通せるか**」で評価する 7 項目に答えられる状態を「本気の B2B」と定義する：

| # | 判断項目 | v3 状態 | v4 目標 |
|---|---|---|---|
| 1 | 契約書（MSA / DPA / SLA） | なし | 雛形あり、法務レビュー済 |
| 2 | セキュリティ質問書回答（CAIQ-Lite / VSA） | なし | テンプレ即提出可 |
| 3 | 認証統合（SSO / SAML / SCIM） | 個人 OAuth のみ | SAML 2.0 + SCIM 2.0 |
| 4 | 監査ログ・データ保持・削除ポリシー | 部分 | 全アクション + 7 年保持 + 完全削除 API |
| 5 | 99.9% SLA + ステータスページ | なし | 公開ステータスページ + クレジット規定 |
| 6 | 請求書払い / PO 対応 / NET30 | Stripe Checkout のみ | 手動請求 + Stripe Invoicing 両対応 |
| 7 | データ所在 / FERPA / GDPR / SOC2 ロードマップ | 未整理 | DPA + データマップ + ロードマップ公開 |

v4 はこの 7 項目を **Phase 単位で全部塗りつぶす**ことを目標にする。

## 2. 法務・契約パッケージ（v4 で新設）

### 2.1 配布する契約書一式

すべて `legal/` ディレクトリ配下に英文 + 日本語訳を配置：

```
legal/
├── MSA.md                    # Master Services Agreement
├── DPA.md                    # Data Processing Agreement (GDPR + UK GDPR)
├── BAA.md                    # Business Associate Agreement (HIPAA、必要校のみ)
├── SLA.md                    # Service Level Agreement
├── AUP.md                    # Acceptable Use Policy
├── SUBPROCESSORS.md          # サブプロセッサ一覧（Supabase, OpenAI, Stripe, etc.）
├── SECURITY_OVERVIEW.md      # 営業同行用の 1 ページャ
├── CAIQ-Lite_Lecsy.xlsx      # CSA CAIQ-Lite 回答済テンプレ
├── VSA_Lecsy.xlsx            # ベンダーセキュリティアセスメント回答済
├── SOC2_Roadmap.md           # SOC2 取得ロードマップ
└── INSURANCE.md              # サイバー保険の付保証明
```

### 2.2 SLA の中身

```
Uptime:
  Business:    99.9%   (月 43分のダウンタイム許容)
  Enterprise:  99.95%  (月 21分のダウンタイム許容)

Credit:
  < 99.9%: 月額の 10% クレジット
  < 99.0%: 月額の 25% クレジット
  < 95.0%: 月額の 50% クレジット

定義:
  ダウンタイム = ステータスページで "incident" 状態の累積時間
  ただし、計画メンテナンス・iOS 単独動作可能時間・顧客起因は除外

報告:
  ステータスページ: status.lecsy.app
  インシデントは 15 分以内に初報、1h 毎に更新、解決後 5 営業日で RCA
```

### 2.3 DPA の主要条項

- **データ管理者:** 顧客（学校）
- **データ処理者:** Lecsy
- **サブプロセッサ:** Supabase (DB/Auth)、OpenAI (AI)、Stripe (課金)、Resend (メール)、Sentry (エラートラッキング)、Cloudflare (CDN/WAF)
- **データ所在:** デフォルト US-East、Enterprise は EU-West / Asia-Pacific 選択可
- **保持期間:** 契約終了後 30 日でハードデリート、要求あれば 7 日に短縮
- **越境転送:** EU 顧客向けは SCC 2021 + 補足措置
- **インシデント通知:** 72 時間以内
- **監査権:** Enterprise は年 1 回、Business は CAIQ + SOC2 報告書で代替

## 3. 認証統合（SSO / SAML / SCIM）

### 3.1 個人ログイン（既存）

- Apple / Google OAuth (B2C 互換)
- Magic Link メールログイン

### 3.2 SAML 2.0 SSO（Business 以上）

- IdP 想定: Okta / Microsoft Entra ID (Azure AD) / Google Workspace / OneLogin / JumpCloud / Shibboleth (大学)
- **SP 設定 UI**: `/org/[slug]/sso` で管理者がメタデータ XML をアップロード
- **JIT プロビジョニング**: SAML Assertion から `email`, `firstName`, `lastName`, `groups` を取り `organization_members` を active 作成
- **属性マッピング**: グループ `lecsy-teacher` → role `teacher` 等の対応表を組織側で設定
- **ドメイン強制**: SAML 有効組織は `allowed_email_domains` のメアドのみ受け入れる
- **強制 SSO モード**: パスワード/OAuth ログインを禁止し、SSO のみ受け付ける設定

### 3.3 SCIM 2.0（Enterprise）

- エンドポイント: `https://api.lecsy.app/scim/v2/`
- 対応リソース: `Users`, `Groups`
- 対応操作: GET / POST / PATCH / DELETE
- グループ → ロール / クラス自動連動
- 退職処理: SCIM の `active=false` で organization_members を `removed` に
- IdP 認定: Phase 2 で Okta OIN 申請、Phase 3 で Azure Gallery 申請

### 3.4 多要素認証

- B2B owner / admin は **MFA 必須化オプション**を組織設定で有効化可能
- TOTP（Authenticator アプリ）標準対応
- WebAuthn / Passkey は Phase 2

## 4. データガバナンス

### 4.1 データ分類

| 分類 | 例 | 暗号化 | アクセス |
|---|---|---|---|
| **P0 機密** | 録音音声、文字起こし、メアド、SSO トークン | at-rest AES-256, in-transit TLS 1.2+ | 本人 + role-based |
| **P1 内部** | 利用統計、AI 生成物 | 同上 | 同組織 admin+ |
| **P2 公開** | 組織名、プラン、ロゴ | TLS のみ | 組織内 |
| **P3 ログ** | 監査ログ、Sentry エラー | 同上 | 自社 + 顧客 |

### 4.2 データマップ

```
[iPhone] ──録音──→ [WhisperKit on-device] ──テキスト──→ [Supabase Postgres (US-East)]
                                                            │
                                                            ├──→ [OpenAI API (要約)]
                                                            ├──→ [Stripe (課金メタのみ)]
                                                            └──→ [Sentry (エラー)]

音声ファイル: 端末内のみ。クラウドには送らない（オプションで送信可能）
文字起こし: Supabase に保存
```

このマップを `legal/SECURITY_OVERVIEW.md` に図入りで掲載。

### 4.3 保持期間ポリシー

| データ | デフォルト保持 | 顧客設定可 |
|---|---|---|
| 録音テキスト | 永続 | 90日 / 1年 / 3年 / 永続 |
| 音声ファイル | 端末のみ（クラウド未保存） | クラウド保存 ON 時は同上 |
| 監査ログ | 7 年（FERPA 準拠） | 短縮不可（コンプラ要件） |
| 利用統計 | 永続（集計済） | 個人特定不可化済 |
| ログイン履歴 | 1 年 | 短縮不可 |

### 4.4 削除権 (Right to be Forgotten)

- ユーザー自身: 設定から「アカウントを削除」で 30 日以内にハードデリート
- 組織管理者: メンバー削除時にデータの扱いを 3 択
  - **Detach（既定）**: 個人録音は本人のまま、組織共有録音は組織保持
  - **Anonymize**: 共有録音から個人識別子除去
  - **Hard Delete**: 共有録音も完全削除（要 owner 承認）
- GDPR Art.17 リクエストは 30 日以内に完了

## 5. セキュリティ仕様

### 5.1 アプリケーション

- 全エンドポイント TLS 1.2+ 強制（HSTS preload）
- CSP: `default-src 'self'`, script-src nonce 方式
- 全 API で Origin / Referer + CSRF token 二重チェック
- レート制限: ユーザー 100 req/min, 組織 1000 req/min, IP 500 req/min
- WAF: Cloudflare Pro
- 依存脆弱性: GitHub Dependabot + npm audit を CI で必須化
- 静的解析: Semgrep + ESLint security ruleset
- ペネトレーションテスト: 年 1 回（Phase 2）

### 5.2 インフラ

- Supabase: SOC2 Type II 取得済（サブプロセッサとして証跡あり）
- DB バックアップ: 日次 + PITR 7 日（Business）/ 30 日（Enterprise）
- DR 訓練: 半期に 1 回、復旧目標 RTO 4h / RPO 1h
- 鍵管理: Supabase Vault + 環境変数は GitHub Secrets / Vercel Encrypted Env
- アクセス制御: 自社メンバーは MFA 必須、最小権限、四半期レビュー

### 5.3 アプリ層認可（v4 で硬化）

- 全 Edge Function で `is_super_admin()` / `is_org_role_at_least()` を **最初の 5 行以内に呼ぶ**
- RLS は防御の最終層、アプリ層を貫通させない
- 全 PR で「権限チェックの行番号」を記載するテンプレを CODEOWNERS に設定

### 5.4 Apple / OS セキュリティ

- App Transport Security: 例外なし
- iOS Keychain: アクセストークンを `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`
- Privacy Manifest: 既存 `PrivacyInfo.xcprivacy` 維持
- ATT: 録音データを広告/トラッキングに使わないため不要だが、明示的に「使わない」と Privacy Nutrition Labels に記載

## 8. 課金・調達対応

### 8.1 課金経路

| 経路 | 対応セグメント | UI |
|---|---|---|
| Stripe Checkout（カード / ACH） | SMB | Web `/org/[slug]/billing` |
| Stripe Invoicing（請求書 + NET30） | MID | 同上、Web で「請求書払いに切替」 |
| 手動請求書 + 銀行振込 | ENT | Web `/admin` で発行、PDF 添付 |
| PO（Purchase Order）対応 | MID / ENT | Web で PO 番号を入力、検収後課金開始 |

### 8.2 価格交渉ガードレール

- 公式割引: 年契 -15%、2 年契 -20%、3 年契 -25%
- ボリューム: 500 席以上 -10%、1000 席以上 -15%
- 教育機関割引: -10%（重ね掛け不可）
- 学期払い: 年契を 2 回分割可能、追加料金なし

### 8.3 解約条件

- Starter / Growth: 月次解約可、当月分まで請求
- Business: 解約は次回更新時、3 ヶ月前通知
- Enterprise: MSA に従う（通常 60 日前通知）

### 8.4 Apple App Store 規定との整合

- B2B 組織課金は **Reader app 例外（3.1.3(b)）+ 法人取引** で IAP 対象外
- iOS では「アップグレード」ボタンが Stripe へ遷移（外部 URL は許容）
- B2C 個人 Pro は引き続き IAP
- App Review Notes に「This app supports both consumer (IAP) and business (invoiced) tiers per guideline 3.1.3(b)」と明記

#### M15. SCIM トークン
```sql
CREATE TABLE org_scim_tokens (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),

CREATE INDEX idx_scim_org ON org_scim_tokens(org_id) WHERE revoked_at IS NULL;
```

```sql
  PRIMARY KEY (org_id, snapshot_date)
);
```

毎日 00:00 UTC に Edge Function `org-seat-snapshot` で全組織の席数を記録。請求は月末の最大値で行う（true-up）。

## 13. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| 創業者 1 人体制での SLA 維持 | 高 | Phase 1.5 までは 99.9% を best-effort、Phase 2 で副業エンジニア追加 |
| Apple 審査で B2B 課金が IAP 強制とされる | 高 | Reader app 例外を Review Notes で予告、ダメなら StoreKit 切替の二段構え |
| Supabase 障害 | 中 | RTO 4h、iOS 単独動作で部分稼働、Phase 3 でマルチリージョン |
| OpenAI 障害 | 中 | AI 機能は graceful degradation、録音/文字起こしは継続 |
| FERPA 違反のリスク | 高 | DPA 必須化、生徒データは Detach 既定、米国弁護士レビュー |
| 大学の長い調達サイクル | 中 | Phase 1.5 で SMB を集め、Phase 2 で MID、Phase 3 で ENT という順序 |
| SOC2 取得遅延 | 中 | Vanta で半自動化、Phase 2 中に Type I、Phase 3 中に Type II |
| 競合（Otter, Microsoft Copilot）参入 | 高 | オンデバイス + 多言語 + 教育特化で差別化、価格は競合の 1/3 |
