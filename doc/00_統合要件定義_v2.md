# Lecsy 統合要件定義書 v2（iOSアプリ単体・B2C/B2B統合）

**作成日:** 2026-04-06
**ステータス:** 確定版（これ以降の実装はこの文書に従う）
**置き換え対象:**
- `doc/01_要件定義書.md`（B2C単体時代の旧版）
- `B2B/B2B設計/管理画面_要件定義書.md`（Web管理画面案・破棄）
- `B2B/現場フロー_統合アプリ.md` の Web 管理画面に関する記述

> **方針サマリ:** Web 管理画面は作らない。**iOS アプリ 1 本で B2C と B2B の両方をカバー**する。組織機能はロールに応じて同じアプリ内に出現する。Web は LP（マーケ）と Stripe Webhook 受信、招待リンク受諾エンドポイントのみに限定する。

---

## 0. なぜこの方針か

| 旧案（破棄） | 新案（採用） |
|---|---|
| iOS（B2C）+ Web 管理画面（B2B）の二本立て | **iOS 単一アプリ + 軽量 Web（LP/Webhook のみ）** |
| 学校の管理者は PC ブラウザで操作 | 管理者も iPhone/iPad の Lecsy アプリで操作 |
| UI/UX を 2 系統メンテ | UI 1 系統。SwiftUI のみ |
| OAuth・課金・RLS を 2 環境で検証 | 検証は iOS フローに集約 |

**採用の根拠**
1. ターゲット（語学学校・IEP のディレクター/講師）は授業中に iPad/iPhone を持ち歩く。PC 前に座る時間より授業現場の方が長い。
2. 営業デモが「**生徒が使うアプリの中に管理画面が出てくる**」という 1 ストーリーで完結する。
3. 個人開発者として保守コストを最小化（Next.js 管理画面分の工数 ~0）。
4. App Store 1 本のみで OK。Web 管理画面側の認証/CSRF/XSS 対策が不要になる。

## 1. プロダクト全体像

### 1.1 ユーザー区分（同一アプリ内）

| 区分 | 入口 | 課金 | 主機能 |
|---|---|---|---|
| **B2C 個人ユーザー** | App Store DL → 個人ログイン | 個人 Pro サブスク（任意） | 録音/文字起こし/AI要約 |
| **B2B 組織メンバー（student）** | 個人としてログイン → メール照合で自動所属 | 組織課金で全機能解放 | 上記 + 組織用語集/組織共有 |
| **B2B 組織講師（teacher）** | 同上 | 組織課金 | 上記 + クラス管理/Glossary生成 |
| **B2B 組織管理者（admin/owner）** | 同上 | 組織課金 | 上記 + メンバー管理/課金/設定 |
| **スーパー管理者（自分のみ）** | 専用メアドでログイン | n/a | 全組織の作成/権限付与/監査 |

**重要原則:**
- アプリは 1 本。区分の差は **UI の出し分けのみ**。
- 個人ユーザーが組織に追加されたら、同じアプリのまま組織機能タブが出現する。
- 組織から外れても、個人としての B2C 利用は継続できる（データは残る）。
- アプリの再インストールは一切不要。

### 1.2 採用しないこと（明確な NO）

- ❌ Web 管理ダッシュボード（`/org/[slug]` ルート群は廃止）
- ❌ 「B2B 専用アプリ」を別途リリースすること
- ❌ Web で組織を作成・メンバー追加する UI
- ❌ 招待メール自動送信（Phase 1 では Deep Link 共有で代用）
- ❌ ロゴアップロード、SSO/SAML、LMS 連携（Phase 2 以降）

### 1.3 Web 側の最小残存スコープ

| 残す | 理由 |
|---|---|
| `web/app/page.tsx`（LP） | App Store への送客 |
| `web/app/privacy`, `terms` | App Store 審査要件 |
| `supabase/functions/stripe-webhook` | Stripe からの Webhook 受信に Edge Function が必要 |
| `https://lecsy.app/invite/[token]` の **薄い Universal Link 受け口**（iOS が未インストールの人を App Store に誘導 → 起動後にトークンを引き継ぐ） | 招待 UX |

これ以外の web ルート（`/app`, `/org/**`, `/api/org/**`）は **削除**する。

## 2. iOS アプリの IA（情報設計）

### 2.1 タブ構成

#### 個人ユーザー（組織未所属）
```
[ホーム]    録音
[ライブラリ] 自分の録音一覧
[設定]      アカウント / Pro / 言語 / プライバシー
```


### 2.3 スーパー管理者専用 UI（自分だけ）

設定タブの最下部に「Super Admin」セクションを出す。判定は **クライアント側のメアドチェックのみでは不十分**なので、サーバ側でも `is_super_admin()` を必ず再評価する（`super_admin_emails` テーブル + RLS）。

スーパー管理者画面で出来ること：
- 組織を新規作成（name/slug/type/plan/max_seats）
- 任意のメアドを任意組織に owner として直接追加（**営業現場で iPad から実施**）
- 全組織の一覧・利用状況閲覧
- 組織の停止/削除
- 監査ログの全件閲覧

> これにより「私が認証したアカウントだけに権限を渡す」要件は **スーパー管理者経由でのみ owner 権限が発生する**という形で物理的に保証される。

## 3. 認証と組織所属モデル

### 3.1 認証

- Supabase Auth（既存）
- プロバイダ: Apple / Google
- メール正規化: 保存・比較とも `lower(trim(email))`

### 3.2 組織への所属モデル

招待リンク方式は **使わない**（漏れのリスク・運用負荷）。代わりに **メール一致による pending → active 自動昇格**方式を採用する（既に `20260401400000_remove_invites_direct_member_add.sql` で導入済み）。

```
[admin が iPad で生徒のメアドを追加]
        ↓
organization_members に row 作成（user_id=NULL, status='pending', email=入力値）
        ↓
[本人が App Store から Lecsy をインストール → ログイン]
        ↓
ログイン直後に activate_pending_memberships(uid, email) を呼ぶ
        ↓
pending row の user_id を埋め、status='active' に更新
        ↓
アプリのタブが再構成され「組織」タブが出現
        ↓
「[組織名] に参加しました」トースト
```

**厳格なメール一致を強制する制約:**
- DB: `activate_pending_memberships(p_user_id, p_email)` は `email = lower(p_email)` で照合（実装済み）
- iOS: 呼び出し時に必ず `auth.user.email` をそのまま渡す（ユーザー入力は受けない）
- ドメイン制限が必要な学校向けには、`organizations.allowed_email_domains TEXT[]` を Phase 1.5 で追加し、許可ドメイン外は pending 作成自体を拒否する RLS を入れる

### 3.3 マルチ組織対応

1 ユーザーが複数組織に所属できる（既存 schema が許容）。アプリ側は `currentOrganizationId` を `@AppStorage` に保持し、設定タブで切替可能にする。録音は常に「現在選択中の組織」のコンテキストで作成される。組織未選択時は個人録音扱い。

## 4. データモデル

### 4.1 既存テーブル（変更なし／微修正）

- `transcripts` — 個人所有のまま。`organization_id UUID NULL` と `visibility TEXT` を **追加**する（マイグレーション必要）。
- `summaries`, `usage_logs`, `rate_limit_logs`, `subscriptions`, `reports` — 変更なし
- `organizations`, `organization_members` — 既に作成済み
- `organization_invites` — **既に DROP 済み**。復活させない。
- `super_admin_emails`（or 関数 `is_super_admin()`） — 既に存在

### 4.2 新規追加マイグレーション（v2 で必要なもの）

#### M1. transcripts に組織コンテキストを追加
```sql
ALTER TABLE transcripts
  ADD COLUMN organization_id UUID NULL REFERENCES organizations(id) ON DELETE SET NULL,
  ADD COLUMN visibility TEXT NOT NULL DEFAULT 'private'
    CHECK (visibility IN ('private', 'class', 'org_wide'));

CREATE INDEX idx_transcripts_org ON transcripts(organization_id) WHERE organization_id IS NOT NULL;
```

RLS 追加:
```sql
-- 同じ組織の teacher/admin/owner は visibility >= 'class' を閲覧可
CREATE POLICY "transcripts_select_org_staff" ON transcripts FOR SELECT USING (
  organization_id IS NOT NULL
  AND visibility IN ('class', 'org_wide')
  AND EXISTS (
    SELECT 1 FROM organization_members om
    WHERE om.org_id = transcripts.organization_id
      AND om.user_id = (SELECT auth.uid())
      AND om.status = 'active'
      AND om.role IN ('teacher', 'admin', 'owner')
  )
);
```

#### M2. classes / class_members
```sql
CREATE TABLE org_classes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  language TEXT NOT NULL DEFAULT 'en',
  semester TEXT,
  teacher_id UUID REFERENCES auth.users(id) ON DELETE SET NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE org_class_members (
  class_id UUID NOT NULL REFERENCES org_classes(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  PRIMARY KEY (class_id, user_id)
);
```
RLS: 同組織の active メンバーのみ閲覧可、teacher 以上が自分のクラスを更新可。

#### M3. glossary_entries
```sql
CREATE TABLE org_glossary (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  term TEXT NOT NULL,
  definition TEXT NOT NULL,
  example TEXT,
  target_language TEXT NOT NULL,
  translation TEXT,
  difficulty TEXT CHECK (difficulty IN ('beginner','intermediate','advanced')),
  source_transcript_id UUID REFERENCES transcripts(id) ON DELETE SET NULL,
  created_by UUID REFERENCES auth.users(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(org_id, term, target_language)
);
```
RLS: 組織 active メンバーは閲覧可、teacher 以上が INSERT/UPDATE。

#### M4. audit_logs
```sql
CREATE TABLE audit_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id UUID REFERENCES organizations(id) ON DELETE CASCADE,
  actor_user_id UUID REFERENCES auth.users(id),
  action TEXT NOT NULL,
  target_type TEXT,
  target_id TEXT,
  metadata JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_audit_org_time ON audit_logs(org_id, created_at DESC);
```
admin 以上のみ SELECT、INSERT は SECURITY DEFINER 関数経由のみ。

#### M5. seats 上限の DB トリガー（race condition 対策）
```sql
CREATE OR REPLACE FUNCTION enforce_seat_limit() RETURNS TRIGGER AS $$
DECLARE
  active_count INT;
  seat_limit INT;
BEGIN
  SELECT max_seats INTO seat_limit FROM organizations WHERE id = NEW.org_id;
  SELECT count(*) INTO active_count
    FROM organization_members
    WHERE org_id = NEW.org_id AND status IN ('active','pending');
  IF active_count >= seat_limit THEN
    RAISE EXCEPTION 'seat_limit_exceeded';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_enforce_seats
  BEFORE INSERT ON organization_members
  FOR EACH ROW EXECUTE FUNCTION enforce_seat_limit();
```

#### M6. members RLS 再帰の解消
現在 `members_select` ポリシーが `organization_members` を自己参照しており、Postgres の policy 再帰で詰まる可能性がある。`SECURITY DEFINER` ヘルパに切り出す:
```sql
CREATE OR REPLACE FUNCTION is_org_member(p_org UUID, p_user UUID)
RETURNS BOOLEAN LANGUAGE sql SECURITY DEFINER STABLE AS $$
  SELECT EXISTS(
    SELECT 1 FROM organization_members
    WHERE org_id = p_org AND user_id = p_user AND status = 'active'
  );
$$;
```
ポリシーは `USING (is_org_member(org_id, auth.uid()))` に書き換える。

### 4.3 マイグレーション順序

```
M6 → M5 → M1 → M2 → M3 → M4
```
（M6 を先にして既存ポリシーを安全化してから残りを足す）

### 5.4 課金（B2B）

- B2B プランの Stripe Price は別管理:
  - `price_b2b_starter`: $299/月
  - `price_b2b_growth`: $599/月
  - `price_b2b_enterprise`: 手動
- iOS から「アップグレード」をタップ → `create-org-checkout-session` Edge Function を呼ぶ → 戻った URL を `SFSafariViewController` で開く → Stripe で決済 → Webhook で `organizations.stripe_subscription_id` を更新
- **App Store ガイドライン 3.1.3(b)** に注意:
  - B2B SaaS 契約（管理者が組織のために購入）は IAP 強制対象外（"Reader app" / 企業向けの除外）
  - 個人 B2C Pro サブスクは **既存どおり StoreKit / IAP** を使う
  - この線引きを `OrgBillingView` に明示し、画面内で「これは組織管理者のための法人向け契約です。アップグレードはブラウザで行います」と説明

### 5.5 Pro 機能の解放判定

```swift
var isPro: Bool {
  hasActivePersonalSubscription
  || currentMembership?.organization.plan != nil
}
```
判定はサーバ側 Edge Function `summarize` でも同じロジックで再評価する（既存 `org-ai-assist` がそれに該当）。

## 7. セキュリティ要件

| 項目 | 対策 |
|---|---|
| 権限付与 | owner 権限はスーパー管理者のみが付与可。`grant_ownership(org_id, email)` は SECURITY DEFINER + `is_super_admin(auth.uid())` ガード |
| RLS 再帰 | M6 の `is_org_member()` ヘルパで解消 |
| 座席上限 | M5 のトリガーで race condition を排除 |
| メール一致 | `activate_pending_memberships` で `lower(trim(email))` 厳格一致 |
| 監査ログ | メンバー追加/削除/ロール変更/組織削除/課金変更は `audit_logs` に必ず INSERT |
| owner 保護 | 最後の owner は削除/降格不可（DB トリガーで強制） |
| 個人情報 | 組織メンバーのメアドは teacher 以上にしか見せない |
| iOS ローカル | 個人録音は端末内 + Supabase（暗号化転送）。組織共有時のみ visibility を上げる |
| Stripe Webhook | 既存の署名検証を継続。`metadata.org_id` の有無でルーティング |

## 付録 A. 用語

- **active メンバー**: `organization_members.status = 'active'` かつ `user_id IS NOT NULL`
- **pending メンバー**: admin が追加したがまだ本人がログインしていない状態
- **スーパー管理者**: `super_admin_emails` テーブルに存在するメアドのユーザー（現状: `nittonotakumi@gmail.com` のみ）
- **visibility**: 録音の公開範囲。`private` / `class` / `org_wide`

## 付録 B. 既知のリスクとミティゲーション

| リスク | 対策 |
|---|---|
| 組織の admin が iPhone を紛失 | スーパー管理者が iOS から別 admin を即時付与 |
